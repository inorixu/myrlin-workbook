<task type="auto">
  <name>Task 1: Add preference checking to queuePush and shouldNotify helper</name>
  <files>src/web/push.js</files>
  <action>
1. Add a `shouldNotify(device, eventType)` function:
   - Takes a device record and an event type string
   - Maps event types to preference keys:
     - 'session:complete' -> `pushPreferences.sessionComplete`
     - 'session:needs-input' -> `pushPreferences.sessionNeedsInput`
     - 'conflict:detected' -> `pushPreferences.fileConflicts`
     - 'task:review' -> `pushPreferences.taskReview`
   - If the preference key is not found (unknown event type), default to true (send it)
   - If the device has no pushPreferences object, default to true (send everything)
   - Return the boolean value

2. Update `queuePush(store, notification)` to check preferences:
   - The notification object should include a `type` field (e.g., 'session:complete')
   - When iterating devices, call `shouldNotify(device, notification.type)` before adding to queue
   - Skip devices where shouldNotify returns false
   - Log skipped devices at debug level: `[Push] Skipping ${device.deviceName} for ${notification.type} (preference disabled)`

3. Update all four event handlers in `setupPushListeners` to include the `type` field in their notification objects:
   - Session completed (running -> stopped): `type: 'session:complete'`, `route: '/(tabs)/sessions/${session.id}'`
   - Session needs input: `type: 'session:needs-input'`, `route: '/(tabs)/sessions/${session.id}'`
   - Task ready for review: `type: 'task:review'`, `route: '/(tabs)/tasks'`
   - File conflict: `type: 'conflict:detected'`, `route: '/(tabs)/sessions'`

This ensures every push notification carries a type for preference filtering and a route for deep linking.
  </action>

  </verify>

</task>

1. In push.js `setupPushRoutes(app, requireAuth, getStore)`, add two new endpoints:

   **GET /api/push/preferences**:
   - Extract Bearer token from `req.headers.authorization` (split on space, take second part)
   - Call `store.findDeviceByToken(token)` to find the calling device
   - If no device found, return 404 `{ error: 'Device not found for this token' }`
   - Return `{ preferences: device.pushPreferences }` (the full preferences object)

   **PUT /api/push/preferences**:
   - Extract Bearer token from `req.headers.authorization`
   - Call `store.findDeviceByToken(token)` to find the calling device
   - If no device found, return 404 `{ error: 'Device not found for this token' }`
   - Validate `req.body.preferences` (or `req.body`): must be an object, all keys must be in the known set ['sessionComplete', 'sessionNeedsInput', 'fileConflicts', 'taskReview', 'serverOnline'], all values must be boolean
   - If validation fails, return 400 with descriptive error
   - Merge with existing preferences: `{ ...device.pushPreferences, ...validatedPrefs }`
   - Call `store.updatePairedDevice(device.deviceId, { pushPreferences: merged })`
   - Return `{ preferences: merged }`

2. In server.js, no changes needed since setupPushRoutes is already called with `(app, requireAuth, getStore)`. The new endpoints are added inside the same setup function.

3. Make sure both endpoints use the `requireAuth` middleware (they are inside setupPushRoutes which already wraps them).
