<task type="auto">
  <name>Task 1: Add sendPushWithRetry and stale token cleanup for pairedDevices</name>
  <files>src/web/push.js</files>
  <action>
Replace the existing `sendPush` function internals with a retry-capable version. Keep the same external signature so existing callers still work.

1. Add a `sleep(ms)` helper at the top of the file (simple Promise-based delay).

2. Create `sendPushWithRetry(message, maxRetries = 3)`:
   - Loop from attempt 0 to maxRetries - 1
   - On each attempt, POST to EXPO_PUSH_URL with the single message (as array of one)
   - Parse the response JSON. If `result.data[0].status === 'ok'`, return `{ sent: true }`
   - If `result.data[0].details?.error === 'DeviceNotRegistered'`, clean the stale pushToken from pairedDevices (find device by pushToken using store.getPairedDevices(), then call store.updatePairedDevice(deviceId, { pushToken: null })). Return `{ sent: false, reason: 'unregistered' }`
   - On network/fetch error: if not last attempt, sleep `1000 * Math.pow(2, attempt)` then retry. If last attempt, log and return `{ sent: false, reason: 'max_retries' }`
   - Also handle non-ok HTTP response (response.ok === false): treat as transient, retry with backoff

3. Update the existing `sendPush(store, notification)` function:
   - Instead of sending all messages in one batch POST, iterate over devices and call `sendPushWithRetry` for each device individually
   - Source devices from `store.getPairedDevices()` (the pairedDevices array, NOT the legacy pushDevices)
   - Filter to only devices that have a non-null `pushToken`
   - For each device, build a message object: `{ to: device.pushToken, title, body, data: { ...notification.data, route: notification.route || null }, sound: 'default', badge: notification.badge || 0 }`
   - Use `Promise.allSettled` to send to all devices concurrently (no need to serialize)

4. Ensure the module still exports `sendPush` with the same `(store, notification)` signature. Add `flushPushQueue` to exports (will be needed by Task 2 for testing, export a no-op for now until Task 2 adds the queue).

Do NOT modify setupPushRoutes or setupPushListeners in this task. Those are updated in Task 2.

<task type="auto">
  <name>Task 2: Add push batching queue with 2-second flush window</name>
  <files>src/web/push.js</files>
  <action>
Add a batching layer in front of sendPush so rapid events are coalesced.

1. Create module-level state:
   - `const pushQueue = new Map()` (maps pushToken string to array of pending notification objects)
   - `let flushTimer = null`
   - `const BATCH_WINDOW_MS = 2000`

2. Create `queuePush(store, notification)`:
   - Get all paired devices with non-null pushToken from `store.getPairedDevices()`
   - For each eligible device, add the notification to `pushQueue.get(device.pushToken)` (create array if absent)
   - If `flushTimer` is null, set `flushTimer = setTimeout(() => flushPushQueue(store), BATCH_WINDOW_MS)`

3. Create `flushPushQueue(store)`:
   - Clear flushTimer: `clearTimeout(flushTimer); flushTimer = null`
   - For each `[pushToken, events]` in pushQueue:
     - If events.length === 1: send the single event via sendPushWithRetry with the original title/body/data
     - If events.length > 1: create a summary notification:
       - title: `"${events.length} updates"`
       - body: summarize event types (e.g., "2 sessions completed, 1 needs input")
       - data: `{ type: 'batch', count: events.length, route: '/(tabs)/sessions' }`
       - badge: count of running sessions from store (count sessions with status === 'running')
     - Call `sendPushWithRetry(message)` for each token
   - Clear pushQueue

4. Update `setupPushListeners` to call `queuePush(store, notification)` instead of `sendPush(store, notification)` for all event handlers (session:updated, worktreeTask:updated, conflict:detected).

5. Add running session count helper: `function getRunningSessionCount(store)` that counts sessions where status === 'running'. Use this for the badge field in both single and batch notifications.

6. Update `sendPush` to also use the badge count (call getRunningSessionCount).

7. Update exports: `module.exports = { setupPushRoutes, setupPushListeners, sendPush, queuePush, flushPushQueue }`. The `flushPushQueue` export enables testing (force-flush without waiting 2s). The `queuePush` export enables direct access for event trigger wiring in Plan 10-02.

Do NOT break the setupPushRoutes register/unregister endpoints. Those stay as-is for now (Plan 10-02 adds preference endpoints).
