  <action>
Create `src/web/push.js` as a CommonJS module exporting `setupPushRoutes(app, requireAuth, getStore)`.

**Store changes (src/state/store.js):**
- Add `pushDevices: []` to DEFAULT_STATE (top-level, not inside settings). Each device: `{ token: string, platform: 'ios' | 'android', registeredAt: string }`.
- Add `addPushDevice(device)` method that pushes to `_state.pushDevices` (deduplicating by token), saves, and emits `push:registered`.
- Add `removePushDevice(token)` method that filters out the device, saves, and emits `push:unregistered`.
- Ensure `pushDevices` is preserved during state loading (same pattern as `templates`, `features`).

**Push module (src/web/push.js):**

1. `setupPushRoutes(app, requireAuth, getStore)` registers two routes:

   POST /api/push/register (requireAuth):
   - Body: `{ deviceToken: string, platform: 'ios' | 'android' }`
   - Validates deviceToken is a non-empty string, platform is 'ios' or 'android'
   - Calls `getStore().addPushDevice({ token: deviceToken, platform, registeredAt: new Date().toISOString() })`
   - Returns `{ success: true }`

   POST /api/push/unregister (requireAuth):
   - Body: `{ deviceToken: string }`
   - Calls `getStore().removePushDevice(deviceToken)`
   - Returns `{ success: true }`

2. `setupPushListeners(store)` function that attaches store event listeners:

   Listen to `session:updated` events. Compare previous status to new status:
   - If status changed from 'running' to 'stopped': send push with title "Session completed", body "{session.name} has finished", data: `{ type: 'session', sessionId: session.id }`
   - If session logs contain "needs input" or "waiting for" (case-insensitive last log): send push with title "Input needed", body "{session.name} needs your input", data: `{ type: 'session', sessionId: session.id }`

   Listen to `worktreeTask:updated` events:
   - If status changed to 'review': send push with title "Task ready for review", body "{task.description}", data: `{ type: 'task', taskId: task.id }`

   For conflict detection, listen to the `conflict:detected` event if it exists, or add a simple check: when sessions emit updates, if the server detects file conflicts, send push with title "File conflict", body "Conflict detected in {file}", data: `{ type: 'conflict' }`.

3. `sendPush(store, notification)` function:
   - Gets all device tokens from `store.getState().pushDevices`
   - If no devices, return silently
   - Build Expo push messages array: `{ to: device.token, title, body, data, sound: 'default' }`
   - POST to `https://exp.host/--/api/v2/push/send` with `Content-Type: application/json` and `Accept: application/json`
   - Use Node.js built-in `fetch` (available in Node 18+). If fetch is not global, use the `https` module.
   - Log errors to console but do not throw (push is best-effort)
   - Handle Expo's response: if any ticket has `status: 'error'` with `details.error: 'DeviceNotRegistered'`, call `store.removePushDevice(token)` to auto-clean stale tokens

Export: `{ setupPushRoutes, setupPushListeners, sendPush }`
