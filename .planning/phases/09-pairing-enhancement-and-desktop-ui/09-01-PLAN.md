<task type="auto">
  <name>Task 1: Add URL detection helpers and enhance pairing-code endpoint</name>
  <files>src/web/pairing.js</files>
  <action>
Add `os` require at top of pairing.js.

Add two helper functions before the route setup:

1. `detectLanIP(port)` - Iterates `os.networkInterfaces()`, finds first non-internal IPv4 address, returns `http://{address}:{port}` or null.

2. `detectTailscaleIP(port)` - Same iteration but specifically looks for addresses in the 100.x.x.x range (Tailscale CGNAT). Returns `http://{address}:{port}` or null. If multiple 100.x addresses, pick the first.

3. `detectAllUrls(port, store)` - Returns an object with all detected URLs:
   - `local`: always `http://localhost:{port}`
   - `lan`: from detectLanIP
   - `tailscale`: from detectTailscaleIP
   - `tunnel`: from `store.state.settings?.tunnelUrl || null`
   - `custom`: from `store.state.settings?.externalUrl || null`

Export `detectAllUrls` so server.js can reuse it for the server-info endpoint.

Modify GET /api/auth/pairing-code handler:
- Call `detectAllUrls(port, store)` where `port` comes from `req.socket.localPort` (the listening port)
- Build `primaryUrl` as: custom || tunnel || lan || local
- Change `qrPayload` to include `urls` object and `primaryUrl` instead of a single `url`
- Keep backward compat: also include `url: primaryUrl` for older mobile clients

The `getStore` dependency is already passed via `setupPairing` deps. Use `getStore()` to access store.state.settings.

POST /api/auth/pair already returns deviceId and capabilities (added in Phase 8). Verify it also includes the `aiSearch` capability flag by checking if process.env.ANTHROPIC_API_KEY is set. Currently capabilities.aiSearch is missing; add it: `aiSearch: !!process.env.ANTHROPIC_API_KEY`.
