# Phase 13, Plan 01: Error Middleware, Structured Codes, and API Versioning

## Objective

Add consistent error response standards across all mobile-facing API endpoints: structured error format with machine-readable codes, Retry-After header on rate-limited responses, and X-API-Version header on every response.

## Context

- @src/web/server.js - Express app, all API routes, middleware chain
- @src/web/auth.js - Rate limiting logic, auth error responses
- @docs/plans/2026-03-29-server-mobile-support.md Section 5 - Error format spec

## Tasks

### Task 1: Add X-API-Version middleware, structured error helper, Retry-After on rate limits
type="auto"

**Changes:**

1. **X-API-Version middleware** (server.js): Add `app.use` middleware early in the chain that sets `X-API-Version: 1` on every response.

2. **structuredError helper** (server.js): Create a `structuredError(res, statusCode, errorCode, message, retryable)` function that returns `{ error, code, message, retryable }` JSON.

3. **Retry-After header** (auth.js): When rate limiting triggers (429 responses), calculate seconds remaining until window resets and include `Retry-After` header.

4. **Update key error responses** in auth.js to use structured format (401 Unauthorized, 429 Rate Limited, 403 Forbidden).

5. **Update requireAuth 401** to use structured error shape.

**Done criteria:**
- Every API response has X-API-Version: 1 header
- Rate-limited 429 responses have Retry-After header with seconds remaining
- Auth error responses (401, 403, 429) use structured format with machine-readable code
- All 52 existing tests still pass
