<objective>
Add scrollback pagination, session logs pagination, and token refresh endpoint so mobile clients can efficiently fetch terminal output and silently renew authentication.

Purpose: Terminal scrollback can be 100KB+; mobile should fetch only last N lines. Session logs need pagination for long-running sessions. Token refresh prevents forced re-pairing when tokens near expiration.
Output: Three new API endpoints: GET /api/sessions/:id/scrollback, enhanced GET /api/sessions/:id/logs, POST /api/auth/refresh.
</objective>

<task type="auto">
  <name>Task 1: Scrollback pagination endpoint and logs pagination</name>
  <files>src/web/pty-manager.js, src/web/server.js</files>
  <action>
1. In pty-manager.js, add a method `getScrollbackLines(sessionId, { lines, from })` to PtySessionManager:
   - Get the session's scrollback array (array of raw output strings)
   - Join all scrollback chunks into a single string, split by '\n' to get line array
   - `from` param: 'end' means last N lines (default), numeric means start from that line index
   - `lines` param: how many lines to return (default 100, max 1000)
   - Return `{ lines: string[], total: number, from: number, hasMore: boolean }`
   - If session not found or no scrollback, return `{ lines: [], total: 0, from: 0, hasMore: false }`

2. In server.js, add GET /api/sessions/:id/scrollback (requireAuth):
   - Query params: `lines` (default 100, max 1000), `from` (default 'end', or numeric index)
   - Get ptyManager reference (already available in server.js scope)
   - Call ptyManager.getScrollbackLines(req.params.id, { lines, from })
   - Return the result object
   - If no PTY session exists, return `{ lines: [], total: 0, from: 0, hasMore: false }` with 200 (not 404, since session may exist but PTY is dead)

3. In server.js, add GET /api/sessions/:id/logs (requireAuth):
   - Query params: `limit` (default 50, max 100), `offset` (default 0)
   - Get session from store: `store.getSession(req.params.id)`
   - If not found, return 404
   - Get `session.logs` array (default empty)
   - Slice by offset and limit
   - Return `{ logs: slicedLogs, total: session.logs.length, hasMore: (offset + slicedLogs.length) < session.logs.length }`

4. Add JSDoc comments on all new methods and routes.

<task type="auto">
  <name>Task 2: Token refresh endpoint</name>
  <files>src/web/auth.js, src/state/store.js, src/web/server.js</files>
  <action>
1. In store.js, add `refreshDeviceToken(deviceId, newToken, newExpiresAt)`:
   - Find device in pairedDevices by deviceId
   - Update device.token to newToken
   - Update device.expiresAt to newExpiresAt
   - Save state
   - Return updated device or null if not found

2. In auth.js, add the refresh logic within setupAuth(app):
   - Add route: POST /api/auth/refresh
   - Protected by requireAuth (needs valid current token)
   - Rate limit: use isRateLimited with a tighter window (reuse existing rate limiter)
   - Logic:
     a. Extract current token from req.authToken (set by requireAuth)
     b. Find device by token using _getStore: `const store = _getStore(); const device = store.findDeviceByToken(token);`
     c. If no device found, return 403 (token is a browser session, not a device token, refresh not applicable)
     d. Generate new token via generateToken()
     e. Remove old token from activeTokens
     f. Add new token to activeTokens
     g. Calculate new expiresAt: 90 days from now (ISO string)
     h. Call store.refreshDeviceToken(device.deviceId, newToken, newExpiresAt)
     i. Return `{ token: newToken, expiresAt: newExpiresAt, deviceId: device.deviceId }`

3. Export nothing new from auth.js (the route is registered inside setupAuth).

4. Add JSDoc comments on the new route and store method.

IMPORTANT: The refresh endpoint MUST only work for device tokens (found in pairedDevices), not browser login tokens. This is a security boundary: browser tokens are short-lived and tied to the password, device tokens are long-lived and tied to the pairing flow.
